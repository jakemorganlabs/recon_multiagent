Session S01 of the MICT-AGENT-004 build. This document defines the goal, scope, deliverables, build sequence, and acceptance criteria for a single working session. The seven-session arc — Core, Brief & State, Search Agent, Analyst & Coverage, Synthesis & Grounding, Evals & Cassettes, Deploy & Demo — is shown below; each session ends with a clean handoff to the next. This is the portfolio's most ambitious piece before the capstone: three agents, bounded toolsets, a hard-capped coverage loop, deterministic grounding, and the no-action posture that keeps the worst possible outcome a dossier with explicit gaps — never a fabricated claim, never an action on the world.

The half that has no excuse to be wrong, before the agents enter.

This piece has more schemas than any prior portfolio piece (four: Brief, EvidenceItem, Signal, Dossier) and a more complex deterministic backbone — a three-hop citation chain verifier (claim → signal → evidence → snippet present) versus the two-hop checks in RAG-002. Getting all of this right before any agent prompt is written is what keeps the agent sessions (S03–S05) focused on agents instead of debugging schema edges or verifier corner cases.

Same discipline as opened PIPE-001 and RAG-002: no model touches this session. Pure functions, unit-tested into submission, become the stable substrate the orchestration leans on. The coverage decision lives here too — the deterministic check the orchestrator will run after the Analyst to decide whether to loop. The model never decides to loop; this code does.

§1 Goal & Exit Criteria

By the end of this session the following are true. Anything short of this means the session is not done, regardless of how much code was written.

  • Postgres migrations apply cleanly for eight tables (run, brief, evidence, signal, tool_call, dossier, audit, dead_letter). No pgvector — this piece does live web search, not embedding retrieval.
  • All four JSON Schemas validate cleanly with a draft-2020-12 validator.
  • Citation chain verifier is a pure function: given a dossier + signals + evidence, deterministically returns per-claim {verified, recast_as_gap, reason}.
  • Snippet normalization handles whitespace, NBSP, smart quotes, and Unicode NFC — shared between the citation verifier and (later) the grounding gate.
  • Coverage decision is a pure function over signals + brief slots returning {action: loop|proceed, unfilled_required: [...]}; tests cover all-filled, some-unfilled, cap-reached.
  • Idempotency key derivation produces stable hashes for canonicalized requests.
  • CI runs tests + schema validation on push; green on head.

§5 Build Sequence

An ordered set of steps. Each step is small enough to complete and verify before moving to the next; together they cover every deliverable.

  1. Initialize repo. Lock language + lint + test runner. Commit .env.example.
  2. Spin local Postgres via Docker. Do not install pgvector — confirm a plain Postgres is what this piece needs (departure from RAG-002).
  3. Write the eight migrations against §11. Critical fields: evidence.content_hash, evidence.source_url, evidence.snippet, evidence.fetched_at — every claim being re-checkable depends on these existing on every row. signal.evidence_ids as an array. run.request_hash with a unique index for idempotency.
  4. Apply migrations to a fresh DB; verify column types and indexes.
  5. Author brief.schema.json matching §11.2 — strict additionalProperties:false, the nine-slot enum, the target object structure, optional seed_urls and depth.
  6. Author evidence_item.schema.json matching §11.3 — every provenance field required.
  7. Author signal.schema.json matching §11.4 — value, confidence (bounded [0,1]), rationale, evidence_ids array (non-empty unless status is insufficient_evidence).
  8. Author dossier.schema.json matching §11.5 — executive summary + section per slot, every claim citing signal_ids.
  9. Implement snippet normalization. Rules: collapse whitespace runs to single spaces, normalize to NFC, fold smart quotes to ASCII, strip zero-width characters. Document the rules in the file header.
  10. Implement the citation chain verifier as a pure function: verify(dossier, signals, evidence) → per-claim results. Three hops checked in order: (1) cited signal_ids exist and are non-abstaining; (2) cited signals' evidence_ids exist in the evidence set; (3) normalize(snippet) ∈ normalize(evidence_text). On any failure, return {verified:false, recast_as_gap:true, reason}.
  11. Write the citation verifier test suite: passing chains, broken-at-signal (signal not present), broken-at-signal (abstaining signal), broken-at-evidence (evidence not present), broken-at-snippet (snippet not in evidence text), whitespace edge cases, Unicode edge cases.
  12. Implement coverage decision. Inputs: brief, signals. Output: {action, unfilled_required, iteration}. Logic: count signals whose status is insufficient_evidence OR confidence < floor for slots marked required:true. If any AND iteration < cap → action:'loop' with the unfilled slot names; otherwise action:'proceed'.
  13. Write coverage tests: all-filled-proceed, one-unfilled-but-optional-proceed, one-unfilled-required-but-at-cap-proceed, one-unfilled-required-budget-available-loop.
  14. Implement idempotency key: canonicalize request (lowercase, trim whitespace, sort optional fields) then SHA-256. Tests: same canonicalized → same hash; different content → different hash.
  15. Wire CI: tests + schema validation on every push.

§6 Definition of Done

The session ends when every box below is checked. These are stricter than the goal criteria — they include the housekeeping that makes the work durable.

  • git status clean; main protected.
  • Fresh DB + npm run migrate + npm test = green.
  • All four schemas validate cleanly.
  • Citation chain verifier tests cover all four failure modes plus passing case.
  • Snippet normalization handles real-world PDF-extraction dirt (NBSP, smart quotes, ligatures).
  • Coverage decision tests cover all four states.
  • Idempotency: byte-canonicalized request produces stable hash.
  • CI green on head commit.
  • README explains: how to install (no pgvector!), how to run migrations, how to run tests.

§7 Risks & Pitfalls

The mistakes that, in practice, derail sessions of this kind. Each is paired with a concrete mitigation rather than a general warning.

RiskMitigation
Snippet normalization too aggressive vs. too laxSame trade-off as RAG-002 S01: lose too much and verifier false-positives; lose too little and verifier false-negatives. Test against actual web-extraction output, not hand-typed strings.
Citation chain verifier's three-hop logic tangledThree hops × multiple cited ids per claim × cross-field rule (abstaining signals can't be cited) = combinatorial. Write tests before the code; the test suite is the spec.
Coverage decision misreads briefCounting optional slots as required, or missing the iteration cap check, breaks the coverage loop's bound. Type the required flag explicitly; assert iteration ≤ cap in code, not just in expectation.
Migrations omit content_hash or provenance fieldsEvery claim being re-checkable depends on evidence.content_hash, evidence.source_url, evidence.snippet, evidence.fetched_at. Missing any breaks the grounding story irreversibly. Verify against §11.3 line by line.
Idempotency canonicalization too liberal or too strictIf "Northwind Robotics" and " Northwind Robotics " hash differently, duplicate runs sneak through. If "Northwind Robotics" and "Northwind Robotics Inc." hash the same, distinct requests collide. Document the canonicalization rule and test edge cases.
