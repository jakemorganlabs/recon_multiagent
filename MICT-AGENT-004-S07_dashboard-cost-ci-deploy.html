Session S07 of the MICT-AGENT-004 build. This document defines the goal, scope, deliverables, build sequence, and acceptance criteria for a single working session. The seven-session arc — Core, Brief & State, Search Agent, Analyst & Coverage, Synthesis & Grounding, Evals & Cassettes, Deploy & Demo — is shown below; each session ends with a clean handoff to the next. This is the portfolio's most ambitious piece before the capstone: three agents, bounded toolsets, a hard-capped coverage loop, deterministic grounding, and the no-action posture that keeps the worst possible outcome a dossier with explicit gaps — never a fabricated claim, never an action on the world.

Observable, economical, deployed — and visible.

The system works and is measured. This session makes it observable, economical, and deployed — with a demo run that's the portfolio's public face. n8n Cloud is managed, so deployment is mostly configuration and discipline (not infrastructure). The dashboard exposes the §17.3 operating picture; cost tracking with cache stats gives production economics real numbers; the CI release with OIDC-signed provenance is the discipline Piece III earned and the capstone will inherit.

Closing this session ends the Piece IV build arc. The next thing built is the capstone (MICT-FIELD-005), which composes Recon's orchestration onto a real corpus with a human delivery gate. The three-agent, bounded-toolset, schema-gated, deterministic-grounding, no-action structure established in S03–S05 transfers wholesale.

§1 Goal & Exit Criteria

By the end of this session the following are true. Anything short of this means the session is not done, regardless of how much code was written.

  • Every stage emits a structured JSON log line with §17.1 mandatory fields.
  • Run audit includes cost: per-call token counts (input, output, cache_read, cache_creation), summed per run.
  • Monthly cost calculator script aggregates a meaningful number from CI eval runs.
  • Dashboard (Metabase over Postgres recommended) exposes the §17.3 operating picture with 6+ widgets.
  • Health-check endpoint returns 200 without invoking the model.
  • CI release workflow with OIDC-signed provenance attestation — carrying Piece III's discipline forward.
  • n8n Cloud workflow locked down: all parameters externalized, secrets only in n8n credential storage, workflow JSON exports clean of secrets.
  • External smoke test from a non-VPS network: HMAC-signed request → 200 + grounded dossier.
  • Public demo viewable on synthetic Northwind Robotics target, rate-limited, accessible from portfolio site.
  • Runbook: redeploy, rotate secrets, refresh cassettes, restore shared state, check DLQ.

§5 Build Sequence

An ordered set of steps. Each step is small enough to complete and verify before moving to the next; together they cover every deliverable.

  1. Wire the structured logger into every workflow stage. Mandatory fields per §17.1: run_id, stage, status, latency_ms, plus per-stage extras (model_id, tokens, similarity_top, gate_fired). Stages: request_trigger, brief_normalize, search_agent_turns, analyst, coverage_check, synthesis, grounding_gate, persist.
  2. Add cost columns to the audit table (input_tokens, output_tokens, cache_read_input_tokens, cache_creation_input_tokens per model call). Populate from API responses on every Brief / Analyst / Synthesis / Search-Agent-turn call.
  3. Author the date-stamped pricing config matching current Anthropic + search provider rates. Add a runtime warning if the config is older than 90 days.
  4. Build scripts/cost_monthly.sh: aggregate audit rows for the last 30 days; apply pricing; output total cost + per-run cost + cache savings rate (cache_read / (cache_read + cache_creation + uncached_input)).
  5. Set up Metabase over the n8n Cloud Postgres (or a separate read replica if Cloud's Postgres isn't directly queryable). Build the §17.3 widgets one by one: runs over time, status mix, gap rate per slot, coverage iteration rate, evidence/fetch volume, grounding recast rate, cost p50/p95. Document alert thresholds per widget.
  6. Build a lightweight health-check endpoint as an n8n webhook: returns 200 + {db: 'ok', config: 'ok', last_run_age_sec: N} without invoking any model.
  7. Set up external uptime monitor (UptimeRobot recommended for portfolio velocity) pinging the health endpoint hourly. Configure email alert on failure.
  8. Author the CI release workflow at .github/workflows/release.yml: triggered on release published, builds the project, runs evals one more time as a release gate, then publishes a release artifact (e.g., a tagged JSON workflow export + the runbook + the sample eval report) with OIDC-signed provenance attestation. Same shape as Piece III's publish pipeline, different artifact.
  9. Cut a v1.0.0 release; verify the provenance attestation on the release artifact.
  10. Lock down production config: every parameter — budgets, floors, caps, slot taxonomy, allow/deny domains, model ids, pricing — externalized into config files or n8n env vars. Nothing hardcoded. Secrets exclusively in n8n credential storage. grep the exported workflow JSON for any token prefix.
  11. Run the external smoke test: from a laptop on another network, sign an HMAC-authenticated request to the n8n Cloud webhook URL with the synthetic Northwind payload. Verify 200 + grounded dossier returned + dossier persisted in shared state.
  12. Configure the public demo URL. Set up rate limiting (10 requests/hour/IP is generous; tune as needed). Mount the demo URL on the portfolio site.
  13. Run the demo on Northwind Robotics. Render the resulting dossier to markdown (v1.0 — branded PDF is deferred). Make it viewable at the demo URL.
  14. Author docs/runbook.md: redeploy (update the n8n Cloud workflow from the JSON export), rotate HMAC secret (which credential, which restart), refresh cassettes (when, how — also document the API quota implication), restore shared state from backup (the pg_restore path), check DLQ (the SQL query). Be specific — vague runbook entries are worse than no runbook.

§7 Risks & Pitfalls

The mistakes that, in practice, derail sessions of this kind. Each is paired with a concrete mitigation rather than a general warning.

RiskMitigation
Logger noise overwhelms n8n Cloud's log retentionEvery stage emitting verbose JSON can exhaust the log buffer fast. Truncate long fields (evidence text, dossier prose) to first 200 chars in logs. Keep full content in the audit row (different store, different retention).
Dashboard refresh is slow on growing audit tableMetabase queries against the audit table get slower over time. Add indexes on (created_at, status) and (slot_name) from the start. Consider a materialized view for the slowest widgets.
Cost calculator uses stale pricesSame risk as RAG-002 S05: date-stamp the pricing config; warn at runtime if older than 90 days. Better: a small script that diffs local against the current API price page.
Cache stats interpretation reversedcache_read_input_tokens = cache was used (cheap); cache_creation_input_tokens = cache was written (more expensive). The cost math depends on knowing which is which. Read Anthropic docs once carefully; record both separately.
CI provenance complexityPiece III's exact discipline transfers — same workflow shape, same id-token: write permission — but the artifact is different (not an npm package; perhaps a tagged JSON workflow export + the runbook). Decide what "the release artifact" is; document in the runbook.
Demo URL abuse without rate limitingWithout rate limiting, the demo's API quota gets burned by anyone testing it. Rate limit BEFORE going public (1 hour grace period is too long); 10/hour/IP recommended.
Secrets leaking via workflow JSON exportsn8n's exported workflows can inline credentials. Diff the export before commit; grep -r the repo for token prefixes (sk-ant, sk-, etc.) before pushing. Verify automated in a pre-commit hook if possible.
Runbook stubs that aren't actionable"Redeploy: update the workflow" is not actionable. Write the actual click path, CLI command, or screenshot sequence. Test the runbook by performing each procedure once from cold; if anything is unclear, fix it.
Public demo's first impression is a failed evalEval failures on the dashboard or sample report are honest but jarring as a portfolio first impression. Either fix the failures before going public (preferred) or include a clear "known gaps" note alongside the report so the gap is contextual rather than alarming.
