Session S02 of the MICT-AGENT-004 build. This document defines the goal, scope, deliverables, build sequence, and acceptance criteria for a single working session. The seven-session arc — Core, Brief & State, Search Agent, Analyst & Coverage, Synthesis & Grounding, Evals & Cassettes, Deploy & Demo — is shown below; each session ends with a clean handoff to the next. This is the portfolio's most ambitious piece before the capstone: three agents, bounded toolsets, a hard-capped coverage loop, deterministic grounding, and the no-action posture that keeps the worst possible outcome a dossier with explicit gaps — never a fabricated claim, never an action on the world.

Contain the request before letting any agent see it.

The Piece I containment pattern, carried in verbatim: one bounded forced-tool extraction turns the raw request into a schema-valid Brief. An unresolvable target routes to clarify, not to a search. The agents never see the raw request — they only see the validated brief. That's not a UX nicety; it's the boundary that keeps a malformed or adversarial request from steering the downstream pipeline.

This is also the n8n setup session. The workflow scaffold, the authenticated request trigger, the shared-state writes, the idempotency check, the structured logger — all the orchestration plumbing the agent sessions hang from gets stood up here. By the end of the session, the spine exists. Adding agents in S03–S05 is a matter of attaching them to the spine, not building plumbing alongside them.

§1 Goal & Exit Criteria

By the end of this session the following are true. Anything short of this means the session is not done, regardless of how much code was written.

  • n8n Cloud workflow scaffolded with an authenticated request trigger (HMAC, 5-minute replay window).
  • Request → Brief Normalizer (one forced-tool extraction) → Brief schema validation → shared-state write.
  • Unresolvable / ambiguous target → status:'clarify' response, no further processing (AC-1).
  • Idempotency on request_hash: same request posted twice returns the existing run_id, no duplicate run created.
  • Audit row written for every request (including clarify cases).
  • Structured logs emitted on each stage (carrying RAG-002 S05's logger discipline forward).

§2 Prerequisites

Conditions that must be in place before this session can usefully start.

  • S01 deliverables on main; CI green.
  • n8n Cloud workspace available with workflow execution permissions.
  • Anthropic API key with forced-tool-use access.
  • Model decision locked. Recommend claude-haiku-4-5 for the brief extraction — it's a small, contained extraction; cheap haiku is the right tool. (The agents in S03–S05 may use a larger model.)

§3 Scope

In scope

  • n8n workflow with an authenticated request trigger.
  • HMAC verification (5-minute replay window, constant-time signature comparison — carry over the module from RAG-002 S05 if the conventions match).
  • Idempotency check: SELECT * FROM run WHERE request_hash = $1; if exists, return that run_id with 200.
  • Brief Normalizer: forced-tool call with brief.schema.json as the tool definition; temperature 0; pinned model.
  • One-shot repair loop on schema failure.
  • Clarification path: if the model emits {status:'unresolvable'} or target.name fails resolution, return {status:'clarify', reason} without further processing.
  • Shared-state writes: run row at start, brief row on successful extraction.
  • Audit row write at run start (always, regardless of outcome).
  • Structured logger module wired in.

Out of scope

  • Search Agent — S03.
  • Analyst, Synthesis — S04, S05.
  • Web tools — S03.
  • Coverage loop — S04.
  • Grounding gate — S05.

§5 Build Sequence

An ordered set of steps. Each step is small enough to complete and verify before moving to the next; together they cover every deliverable.

  1. Set up the n8n Cloud workspace; create an empty workflow with a webhook (request) trigger.
  2. Wire HMAC verification as the first node after the trigger. Reject unsigned or stale (>5 minute) requests with 401. Use constant-time comparison. Port the module from RAG-002 if the conventions match.
  3. Add the idempotency check: a Code node that reads the request body, computes the request_hash via S01's idempotency.ts, and queries the run table. If found → return existing run_id + 200, stop further nodes. If not found → write the initial run row and proceed.
  4. Add the Brief Normalizer: an LLM node configured for forced tool use with brief.schema.json as the tool's input_schema. tool_choice: {type:'tool', name:'brief_extractor'}. temperature: 0. Pinned model (claude-haiku-4-5 recommended for cost on this contained extraction).
  5. Wire one-shot repair: on schema validation failure, re-call once with the validation error + prior output appended to the system message. On second failure, set the run status to failed, write to DLQ, escalate.
  6. Validate the extracted brief: if target.name is unresolvable OR required slots can't be inferred → set status:'clarify' on the run, write the reason to the audit row, return the clarify response. Otherwise → write the brief to the brief table.
  7. Write the audit row at run start (every request gets one, even those that end in clarify).
  8. Wire the structured logger into every stage. Mandatory fields per §17.1: run_id, stage, status, latency_ms, model_id (where applicable), tokens (where applicable).
  9. Smoke test happy path: post a synthetic request for "Northwind Robotics"; verify a validated brief in brief table with the nine slots.
  10. Smoke test idempotency: post the same request twice; verify only one run row created, second response returns the same run_id.
  11. Smoke test clarification: post a deliberately ambiguous request (e.g., {"target":"Apple"} — which Apple?); verify status:'clarify' returned without further processing.
  12. Smoke test repair loop: temporarily corrupt the schema or the prompt to force a single repair; verify exactly one re-call happens.

§6 Definition of Done

The session ends when every box below is checked. These are stricter than the goal criteria — they include the housekeeping that makes the work durable.

  • Workflow committed; HMAC enforcement verified by signed and unsigned requests.
  • Idempotency verified — same request twice yields one run row.
  • Brief Normalizer produces schema-valid briefs on happy-path requests.
  • Clarification path triggers on unresolvable targets without burning further tokens.
  • Audit row written for every request including clarify cases.
  • Structured log line per stage per request; mandatory fields populated.
  • brief table populated correctly after a happy-path request.

§7 Risks & Pitfalls

The mistakes that, in practice, derail sessions of this kind. Each is paired with a concrete mitigation rather than a general warning.

RiskMitigation
Brief schema too restrictive for reasonable requestsA target with no website, a request with five focus areas, or a domain Anthropic's tokenizer treats oddly — any can break a brittle schema. Test against varied real-world request shapes; loosen the schema when correct, never silently relax validation.
Forced-tool config emits text instead of tool callIf tool_choice is wrong (defaults to auto on some configs), the model may emit prose instead of calling the tool. Pin tool_choice: {type:'tool', name:...} and verify with a known-good call.
HMAC complexity carried over wrong from RAG-002n8n Cloud's request trigger has its own auth idioms — verify the signing format end-to-end with a curl from a different machine. Don't trust that the module ported cleanly without testing.
Idempotency canonicalization brittleWhitespace, case, optional fields, trailing newlines — any of these in the canonicalization rule can let near-duplicate requests through. Document the rule; unit-test edge cases.
Clarification path swallows real errorsA schema failure from a malformed model output is different from an unresolvable target. Distinguish the two; route to clarify only for resolvability issues, not for model failures.
Logger noise overwhelms n8n Cloud retentionEvery stage logging full input/output text can exhaust the log buffer fast. Truncate long fields (brief content, evidence text) to first 200 chars in logs.
