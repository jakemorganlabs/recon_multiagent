Session S04 of the MICT-AGENT-004 build. This document defines the goal, scope, deliverables, build sequence, and acceptance criteria for a single working session. The seven-session arc — Core, Brief & State, Search Agent, Analyst & Coverage, Synthesis & Grounding, Evals & Cassettes, Deploy & Demo — is shown below; each session ends with a clean handoff to the next. This is the portfolio's most ambitious piece before the capstone: three agents, bounded toolsets, a hard-capped coverage loop, deterministic grounding, and the no-action posture that keeps the worst possible outcome a dossier with explicit gaps — never a fabricated claim, never an action on the world.

Read what's there, abstain on what isn't, never decide to loop.

The Analyst is the first agent sealed from the web by toolset construction. Its only tool is read-shared-state; there is no path by which it can fetch a new page, no matter what its prompt says. It reads the evidence the Search Agent gathered, fills each brief slot with a value + confidence + rationale + citing evidence_ids, and abstains (insufficient_evidence) on slots the evidence doesn't support. No support means no answer, ever.

The coverage loop is the orchestrator's job, not the model's. A deterministic Code node inspects the signals, checks brief.slots for required slots that are unfilled or below-floor, and routes either a targeted re-search (one time, with queries scoped to the unfilled slots) or a proceed-to-Synthesis. The iteration counter lives in the run record. The hard cap is structural — at iteration ≥ 1, the controller proceeds regardless. The model never decides to loop.

§1 Goal & Exit Criteria

By the end of this session the following are true. Anything short of this means the session is not done, regardless of how much code was written.

  • Analyst Agent (n8n AI Agent node) bound to read-shared-state tool only — no web tools, verified by workflow JSON inspection (AC-5).
  • For each slot in the brief, Analyst emits a Signal with value, confidence, rationale, evidence_ids.
  • Unsupported slots emit status:'insufficient_evidence' with evidence_ids:[] — no guessing (FR-AN-2, FR-GR-3, AC-4).
  • All signals schema-validated against Signal schema; one-shot repair on failure.
  • Coverage controller (deterministic Code node) inspects required slots; loops once or proceeds.
  • On loop: targeted re-search with queries scoped to unfilled slot names; iteration counter incremented in run record.
  • Hard cap enforced: iteration ≥ 1 → action:'proceed' regardless (AC-8).
  • Smoke test: deliberately thin slot → Analyst abstains → coverage loops once → Analyst fills slot on second pass.

§2 Prerequisites

Conditions that must be in place before this session can usefully start.

  • S03 complete; Search Agent produces evidence for synthetic targets.
  • Anthropic API key (same model recommended as S03 for cache reuse).
  • Test scenario prepared: a synthetic target where one slot (e.g., leadership) has thin first-pass coverage that fills on a targeted re-search.
  • Confidence floor decision locked in config/budgets.json (recommend 0.6 as starting point).

§3 Scope

In scope

  • Read-shared-state tool: a Code node that takes run_id and returns the evidence rows for that run. Bound to Analyst (and later Synthesis) only.
  • Analyst Agent (AI Agent node) with read-shared-state tool only.
  • Tool binding verification — workflow JSON inspection.
  • Analyst system prompt (§15): grounded extractor, fill from evidence only, cite evidence_ids for every non-abstaining signal, abstain when evidence doesn't support.
  • Signal schema enforcement on every emitted signal.
  • One-shot repair loop on schema failure.
  • Coverage controller as a Code node: read signals, check required slots against floor, return {action, unfilled_required, iteration}.
  • Targeted re-search routing: action:'loop' + iteration < cap → re-invoke Search Agent with scoped queries derived from unfilled_required slot names.
  • Iteration counter writes to run.coverage.iterations.
  • Hard cap enforcement: iteration ≥ 1 → proceed regardless of unfilled state.

Out of scope

  • Synthesis Agent — S05.
  • Grounding gate — S05.
  • Eval cassettes — S06.

§5 Build Sequence

An ordered set of steps. Each step is small enough to complete and verify before moving to the next; together they cover every deliverable.

  1. Build the read-shared-state tool: a Code node accepting {run_id} and returning the evidence rows for that run. Inputs validated; outputs schema-bound. This tool will be reused by Synthesis in S05.
  2. Configure the Analyst AI Agent node. Pin the model. temperature 0. Bind only the read-shared-state tool — verify no other tools bound.
  3. Verify the binding structurally: export workflow JSON, inspect Analyst's tools array, confirm exactly one entry (read_shared_state). Record the JSON snapshot for AC-5 evidence.
  4. Author the Analyst system prompt (§15): grounded extractor, read evidence in shared state, fill each slot ONLY from supporting evidence, cite evidence_ids for every filled slot, mark unsupported slots as insufficient_evidence, never guess. Place above the cache breakpoint.
  5. Wire signal writes: every emitted signal validated against Signal schema (from S01). On schema fail → one-shot repair with validation error appended. On second fail → run status failed, DLQ row.
  6. On success → write to signal table keyed on (run_id, slot_name) — idempotent so a repair overwrites rather than appends.
  7. Build the coverage controller as a Code node: reads the run's signals + brief.slots, calls S01's coverage_decision(), returns {action, unfilled_required, iteration}.
  8. Wire the coverage controller into the workflow: positioned after Analyst, before Synthesis.
  9. Build the targeted re-search routing: if action:'loop' AND run.coverage.iterations < cap → increment iteration counter, route back to Search Agent with a scoped brief (the unfilled_required slot names appended as focus).
  10. Enforce the hard cap: if iteration ≥ cap → set action:'proceed' regardless of unfilled state (this is the structural enforcement of AC-8 / FR-OR-3).
  11. Smoke test on Northwind synthetic: run from a clean state; verify signals emitted, gaps surfaced as insufficient_evidence, no guesses.
  12. Smoke test coverage loop: configure a deliberately thin slot (mock the Search Agent's first pass to skip queries about a particular slot like leadership); verify Analyst abstains on that slot in pass 1; verify coverage controller fires action:'loop'; verify Search Agent re-runs with scoped queries; verify Analyst fills the slot on pass 2.
  13. Smoke test hard cap: configure a slot that cannot be filled; verify pass 1 abstains, pass 2 abstains, controller fires action:'proceed' regardless on iteration 2, no third loop, gap surfaces.

§6 Definition of Done

The session ends when every box below is checked. These are stricter than the goal criteria — they include the housekeeping that makes the work durable.

  • Analyst Agent produces schema-valid Signals on synthetic Northwind data.
  • insufficient_evidence emitted for unsupported slots; never a guess (verified by spot-checking signals).
  • Read-shared-state tool returns evidence; Analyst has no other capabilities (verified by workflow JSON).
  • Coverage controller fires correctly: loops once on thin-slot scenario, proceeds otherwise.
  • Hard cap enforced — no third loop possible (verified by smoke test).
  • Iteration counter persists to run.coverage.iterations.
  • Workflow JSON snapshot recorded for AC-5 evidence.

§7 Risks & Pitfalls

The mistakes that, in practice, derail sessions of this kind. Each is paired with a concrete mitigation rather than a general warning.

RiskMitigation
Analyst tool binding leakSame risk as Search Agent — must verify by inspecting workflow JSON, not by reading the visual config. The structural enforcement of "no web access" depends on this verification.
Repair loop unboundedIf the repair is implemented by recursion or by re-triggering the same node, two failures can chain indefinitely. Implement via an explicit counter in the Code node wrapper; hard cap at one repair.
Coverage loop unboundedSame risk, same mitigation: the counter lives in run.coverage.iterations, the cap check is in code, the decision is deterministic. Test deliberately with an unfillable slot.
Targeted re-search queries miss the markScoping queries to slot names too literally ("site:* leadership") produces no results. The Search Agent should be allowed to generate variations within the scope — recommend passing the unfilled slot names as focus areas in the brief, letting the agent compose queries.
Confidence floor too highIf nothing clears the floor on plausible evidence, every slot abstains and the loop runs until cap. Set the floor conservatively (recommend 0.6 starting); document the choice; tune after eval.
Repair-and-persist creates duplicate signal rowsIf a failed signal is written first and a repaired signal is written second, you have two rows for one slot. Idempotent write keyed on (run_id, slot_name): the repaired row replaces the failed one.
Coverage controller reads stale signalsRace condition if signals are still being written when the controller queries. Use the same transaction or an explicit completion marker on the Analyst step before the controller starts.

§8 Handoff to the Next Session

S05 picks up signals — some filled, some abstaining — and produces a dossier. The Synthesis Agent (also sealed from the web) narrates the signals; then the deterministic grounding gate (the three-hop verifier from S01) checks every claim before the dossier is returned. S05 closes the end-to-end happy path.
